### 站长前言
{{% warn title="给站长的提醒" %}}
如果你是站长，希望你看完这些思路后能顺手把相关隐患补掉。
{{% /warn %}}

### DNS 相关
#### DNS 历史解析记录
如果目标网站曾直接解析过源站，可以提供一些记录 DNS 历史记录的工具找到源站地址。

{{% tip title="预防方法" %}}
更换 IP 后不要再把域名直接解析到服务器上，最好接入一层 CDN。
{{% /tip %}}

#### 子站扫描
有些 CDN 限制接入数量，可能子站就没接入 CDN，可以通过扫描二级域名来寻找子站的源站地址，大概率也是主站的源站地址。

{{% tip title="预防方法" %}}
把子站也接入 CDN，哪怕只是 Cloudflare 这种基础方案。
{{% /tip %}}

### CDN 相关
#### CDN 解析源站
有些 CDN 在流量包用完时，会解析源站，把量刷完后，可用逼迫其解析源站。

{{% tip title="预防方法" %}}
使用不限量的 CDN，例如 Cloudflare；或者把 CDN 配成达量后直接停止解析。
{{% /tip %}}

### 源站 相关
#### 软件漏洞
例如博客程序的评论回复邮件，在邮件里很容易泄露源站 IP。

{{% tip title="预防方法" %}}
停用此类插件，发送邮件时单独准备一台邮件服务器，或者直接使用成熟的第三方发信服务。
{{% /tip %}}

#### 全网扫描
如果目标服务器有套 CDN，你可以使用一些全网扫描的工具，全网 IPv4 地址扫描。如果某个服务器的 SSL 证书是有效的，且是你要扫描的域名的，那么这台服务器就是此域名的源站服务器。你可以使用 [censys][1] 来实现全网扫描。

{{% caution title="预防方法" %}}
- 设置空证书，避免直接暴露该服务器所对应的域名。
- 设置防火墙，拒绝所有默认入站请求，只对白名单 IP 放行，例如 CDN 节点、搜索引擎和你自己的常用出口 IP。
{{% /caution %}}

#### 关于 CDN 引发的一个小问题
##### 在使用 CDN 后，源站日志均记录为 CDN 节点 IP 的解决方案
如果您也使用 Nginx，那么请在 Nginx 的配置里加入下列参数配置
```shell
nano /etc/nginx/nginx.conf
```
```nginx
    set_real_ip_from   0.0.0.0/0;
    real_ip_header     X-Forwarded-For;
```

[1]: https://censys.io/ipv4
